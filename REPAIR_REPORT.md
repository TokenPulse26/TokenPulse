# TokenPulse Repair Report — 2026-03-28

## Completed Fixes

### ✅ 1. Dashboard Restored (was down)
**Problem:** The web dashboard (`com.tokenpulse.dashboard` launchd service) was in a crash loop. The Python `HTTPServer` was binding to `0.0.0.0` (IPv4 only), but macOS curl defaults to IPv6 (`::1`). Additionally, rapid launchd restarts caused socket race conditions.

**Fix:** 
- Replaced `HTTPServer` with `DualStackHTTPServer` that binds to `::` (IPv6 dual-stack) with `IPV6_V6ONLY=0`, accepting both IPv4 and IPv6 connections
- Added retry logic (5 attempts with 2s delay) for port binding failures
- Added `flush=True` to all print statements for proper logging under launchd

**Result:** Dashboard running and accessible at `http://localhost:4200` (IPv4, IPv6, and auto)

### ✅ 2. OpenClaw Routing Through TokenPulse
**Problem:** OpenClaw config pointed directly at CLIProxy (`http://127.0.0.1:8317/v1`), bypassing TokenPulse. This meant no token tracking was happening for normal usage.

**Fix:** Updated `~/.openclaw/openclaw.json`:
- `cliproxy.baseUrl`: `http://127.0.0.1:8317/v1` → `http://127.0.0.1:4100/cliproxy/v1`

**Result:** All OpenClaw requests now flow through TokenPulse proxy → CLIProxy → Claude. Backup saved at `~/.openclaw/openclaw.json.pre-tokenpulse-reintegration-backup`.

**Note:** The gateway needs to restart to pick up the new config. New sessions and sub-agents will use it automatically.

### ✅ 3. Health Check Endpoint Added
**Problem:** Bare `GET /` on the proxy was forwarded to OpenAI, returning "421 Misdirected Request". This made monitoring and basic connectivity checks confusing.

**Fix:** Added health check handler for `GET /` and `GET /health` that returns:
```json
{
  "status": "ok",
  "service": "tokenpulse-proxy",
  "port": 4100,
  "total_requests_tracked": 2640
}
```

**Result:** Clean health checks for monitoring. Also prevents accidental OpenAI routing for bare requests.

### ✅ 4. Proxy Binary Rebuilt
**Fix:** Rebuilt release binary with health check endpoint included.
